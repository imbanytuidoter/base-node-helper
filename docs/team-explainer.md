## Що це таке?

`base-node-helper` (команда: `bnh`) — це інструмент командного рядка, який обгортає `docker compose` для безпечного та надійного запуску **повної ноди Base blockchain**.

Замість того, щоб вручну запускати `docker compose up`, ви запускаєте `bnh start` — інструмент перевіряє готовність машини, запускає ноду і сповіщає вас якщо щось пішло не так.

**Яку проблему вирішує:**  
Ручний запуск Base-ноди вимагає одночасно відслідковувати багато речей: чи запущений Docker? Чи вільні порти? Чи достатньо місця на диску? Чи ще синхронізується нода? Чи достатньо пірів? Якщо пропустити хоча б одне — нода може відмовити без повідомлення або пошкодити стан. `bnh` автоматизує все це.

## Основний флоу

```
bnh doctor  →  bnh start  →  bnh monitor
```

### Крок 1 — bnh doctor
Запускає 8 preflight-перевірок перед будь-яким стартом:

| Перевірка | Що тестує |
|-----------|-----------|
| Docker запущений | демон docker працює |
| Compose доступний | плагін docker compose встановлений |
| Порти вільні | 30303 (P2P) і 9222 (RPC) не зайняті |
| Місце на диску | мінімум 500 ГБ вільно |
| Швидкість диску | затримка запису менше 200 мкс |
| Синхронізація NTP | системний годинник точний в межах 1 секунди |
| L1 RPC доступний | ваш Ethereum L1 endpoint відповідає |
| Кількість пірів | достатньо пірів видно в мережі |

Кожна перевірка повертає **PASS**, **WARN** або **FAIL**:
- `bnh start` продовжує при PASS і WARN
- `bnh start` зупиняється при будь-якому FAIL і пояснює що виправити

### Крок 2 — bnh start
1. Запускає всі 8 preflight-перевірок
2. Якщо всі пройшли → виконує `docker compose up -d`
3. Звітує які контейнери запустилися

Щоб бачити повний вивід compose — запустіть `bnh start --verbose`.

### Крок 3 — bnh monitor
Працює у фоні (або як systemd-сервіс), опитуючи стан кожні 60 секунд:

- Чи всі контейнери запущені?
- Чи ще синхронізується нода?
- Чи достатньо пірів?

Коли стан змінюється (наприклад, контейнери впали або синхронізація завершилася) — надсилає сповіщення в Discord або будь-який webhook. Сповіщення надходять тільки при **зміні стану**, а не при кожному опитуванні — щоб не засмічувати.

## Всі команди

| Команда | Опис |
|---------|------|
| `bnh init` | Інтерактивне налаштування — створює профіль з шляхом до репо, Discord webhook, налаштуваннями сповіщень |
| `bnh start` | Запускає preflight-перевірки, потім стартує ноду |
| `bnh stop` | Плавне завершення з тайм-аутом 300 секунд |
| `bnh status` | Показує поточний стан контейнерів |
| `bnh doctor` | Запускає повний preflight-звіт без старту ноди |
| `bnh monitor` | Запускає цикл моніторингу здоров'я |
| `bnh upgrade` | Виконує `git pull` у репо ноди; опційно перезапускає ноду |
| `bnh down` | Примусово видаляє все. Потребує прапорці `--force --i-understand`. |

## Встановлення

```sh
curl -fsSL https://base.sh/install | sh
```

Або зібрати з вихідного коду:

```sh
git clone https://github.com/imbanytuidoter/base-node-helper
cd base-node-helper
go build -o bnh ./cmd/bnh
```

**Вимоги:**
- Docker з плагіном Compose
- Go 1.23+ (тільки якщо збираєте з коду)
- Linux, macOS або Windows (WSL)

## Конфігурація

Запустіть `bnh init` один раз щоб створити профіль. Він запитає:

1. **Назву профілю** — "default" підходить для більшості
2. **Шлях до репо ноди** — локальний шлях до `docker-compose.yml` Base-ноди
3. **Discord webhook URL** — для сповіщень (опційно)
4. **Мінімальна кількість пірів** — поріг для сповіщення (за замовчуванням: 5)
5. **Рівень сповіщень** — отримувати тільки `crit`, або ще й `warn`

Конфіг зберігається у `~/.config/base-node-helper/<профіль>.yaml`.

## Як працюють сповіщення

`bnh monitor` відслідковує **зміни стану** (тільки наростаючий фронт):

| Подія | Рівень | Тригер |
|-------|--------|--------|
| Контейнер впав | `crit` | Будь-який контейнер вийшов або впав |
| Нода почала синхронізуватися | `warn` | `eth_syncing` повертає true |
| Кількість пірів впала | `warn` | Кількість нижче вашого мінімуму |

Сповіщення приходить тільки коли проблема **починається**, не при кожному опитуванні.  
Discord-сповіщення показують рівень, заголовок і короткий опис.

## Функції безпеки

**Preflight-шлюз** — `bnh start` не запуститься якщо будь-яка перевірка повертає FAIL. Це не дає запустити ноду на непідготовленій машині.

**Валідація назви профілю** — назви обмежені до `[A-Za-z0-9_-]`. Захист від path traversal атак.

**Обмежене читання HTTP** — всі HTTP-відповіді обмежені до 1 МБ для захисту від витоку пам'яті через шкідливі RPC-ендпоінти.

**Плавне завершення** — `bnh stop` чекає до 300 секунд поки контейнери завершаться коректно.

**Context-aware очікування** — всі цикли очікування реагують на `Ctrl-C` миттєво.

## Архітектура (для розробників)

```
cmd/bnh/           → точка входу
internal/
  cli/             → cobra-команди (init, start, stop, status, doctor, monitor, upgrade, down)
  config/          → завантаження/збереження YAML-профілів, підстановка змінних середовища
  compose/         → виявлення та виконання docker compose
  preflight/       → 8 окремих перевірок здоров'я
  rpc/             → L1 Ethereum RPC клієнт (chain ID, syncing, peer count)
                     Beacon API клієнт (genesis time)
  lockfile/        → shared/exclusive файлове блокування для безпечного паралелізму
  notify/          → відправник Discord і generic webhook
```

**Без зовнішнього стану.** Весь конфіг — у файлах. Немає бази даних, немає фонового демона, якщо ви самі не запустили `bnh monitor`.
